# 阶段一核心基建设计

> 状态更新（2026-04-02）：阶段一已按本设计完成落地，代码已合并到 `main`。实际交付包括 `pkg/config`、`pkg/logger`、`pkg/database`、`verification.md` 以及 TDD 交接文档 `docs/plans/2026-03-18-stage1-core-infra-tdd-spec.md`。本设计文档继续保留为“设计基线”，用于回溯原始方案与后续阶段对照。

## 背景

PureMux 当前仓库处于项目初始化早期阶段，现有内容仅包含 `README.md`、`.gitignore` 与 `.codex/AGENTS.md`，尚未存在任何 Go 模块、业务代码或基础设施实现。

本阶段目标是在严格遵守既定技术栈的前提下，先完成三个核心基础包的设计确认：

- `pkg/config`
- `pkg/logger`
- `pkg/database`

本设计文档只覆盖阶段一基建方案确认，不进入业务实现，不扩展到路由、中间件、统一响应包装或示例业务流。

## 目标

- 建立统一、强类型、可测试的配置加载机制
- 建立全局一致的结构化日志能力，输出 JSON 格式日志
- 建立 MySQL 与 Redis 的标准初始化入口和生命周期管理方式
- 为后续 `cmd/server/main.go` 的组装提供清晰、稳定的依赖装配顺序
- 保持实现风格简单、原生、可维护，避免隐式状态扩散

## 边界

本阶段明确包含：

- `configs/config.yaml` 的配置模型设计
- `pkg/config` 的加载入口与结构设计
- `pkg/logger` 的初始化策略、输出格式与日志切割策略
- `pkg/database` 的 MySQL / Redis 初始化、连接池参数与启动失败策略

本阶段明确不包含：

- HTTP 路由注册
- Huma API 扩展
- 自定义中间件链
- 统一响应包装器
- Model / Repository / Service / Handler 示例业务
- 启动命令与服务监听实现

## 现状事实

- Git 历史仅有初始提交与 `.gitignore` 调整，没有现成 Go 代码可复用
- 仓库内不存在 `go.mod`、`pkg/`、`cmd/`、`configs/`、`docs/plans/` 旧文件
- 当前不存在“终端控件”或其他可参考的渲染实现，相关表述视为任务模板残留，不作为本项目设计依据
- 技术栈已由用户明确锁定，不进行额外选型变更

## 方案对比

### 1. `pkg/config` 加载入口

#### 方案 A：显式加载

由上层启动流程调用 `config.Load(path)`，返回 `*Config` 后再显式传给 logger、database 等依赖方。

优点：

- 依赖关系清晰
- 便于测试注入与替换
- 不依赖包级状态，后续扩展更稳

缺点：

- 调用链上需要显式传参

#### 方案 B：包级单例

由 `config.Init()` 初始化后，通过全局变量或 `config.Get()` 在各处读取。

优点：

- 使用门槛低
- 调用代码更短

缺点：

- 隐式依赖明显
- 测试隔离较差
- 后续多环境或并行测试更容易出现状态污染

#### 最终决策

选择方案 A：显式加载。

### 2. `Config` 结构组织方式

#### 方案 A：分层强类型结构

顶层 `Config` 下拆分以下子结构：

- `App`
- `MySQL`
- `Redis`
- `Log`

配置来源以 `config.yaml` 为主，环境变量作为覆盖层。

优点：

- 结构清晰，和 YAML 映射自然
- 可读性好，适合长期维护
- 后续新增配置项时不易失控

缺点：

- 字段 tag 会相对更多

#### 方案 B：扁平结构

所有字段直接挂在一个大 `Config` 结构体上。

优点：

- 初始写法直接

缺点：

- 字段多后可读性差
- 配置边界不清晰
- 不利于后续模块扩展

#### 最终决策

选择方案 A：分层强类型结构。

### 3. `pkg/logger` 日志切割策略

#### 方案 A：仅依赖 `lumberjack` 默认轮转

使用 `slog` JSON 输出，文件侧接入 `lumberjack`，由文件大小触发轮转，并通过 `MaxAge` 控制保留天数。

优点：

- 实现简单
- 运行稳定

缺点：

- 不能严格满足“按自然日切割”

#### 方案 B：`lumberjack` + 跨日主动轮转

继续使用 `slog` JSON 输出与 `lumberjack`，同时增加一个轻量调度逻辑，在跨午夜后主动调用 `Rotate()`。

优点：

- 在不更换日志库的前提下满足“按天切割”的目标
- 保持技术栈完全一致

缺点：

- 需要额外的后台调度与停止逻辑

#### 最终决策

选择方案 B：`lumberjack` + 跨日主动轮转。

### 4. `pkg/database` 资源装配与启动策略

#### 方案 A：集中式 bootstrap + fail-fast

由 `database` 包统一初始化 MySQL 与 Redis，在启动阶段完成连接参数配置、`Ping` 检查和资源组装，并返回聚合资源对象。

优点：

- 启动顺序清晰
- 配置错误会在启动时立即暴露
- 方便后续在 `main` 中统一关闭资源
- 未来扩展其他基础设施时结构更统一

缺点：

- 启动阶段更严格，坏配置会直接失败

#### 方案 B：分散式单例 / 惰性初始化

MySQL 与 Redis 各自维护包级单例，在业务首次访问时再建立连接。

优点：

- 局部调用更直接

缺点：

- 错误延后到运行时暴露
- 生命周期管理分散
- 不利于脚手架场景下的问题定位

#### 最终决策

选择方案 A：集中式 bootstrap + fail-fast。

## 最终决策汇总

本阶段确认采用以下组合：

- `pkg/config`：显式加载
- `Config` 组织：分层强类型结构
- `pkg/logger`：`slog` JSON + `lumberjack` + 跨日主动 `Rotate()`
- `pkg/database`：集中式 bootstrap + fail-fast

## 设计细化

### `pkg/config`

建议职责：

- 定义 `Config` 顶层结构与四个子配置结构
- 提供单一入口 `Load(path string) (*Config, error)`
- 支持 YAML 文件加载
- 支持环境变量覆盖
- 不负责目录探测、热更新、远程配置或运行时变更

设计原则：

- 只做“读取与解析”，不做依赖初始化
- 默认值、必填项与字段含义通过结构体 tag 明确表达
- 返回值为显式对象，不泄漏包级可变状态

### `pkg/logger`

建议职责：

- 根据 `Config.Log` 初始化 `*slog.Logger`
- 输出格式固定为 JSON
- 支持 `stdout` 与文件输出策略
- 文件输出使用 `lumberjack`
- 如启用文件日志，则补充跨日主动轮转能力

设计原则：

- logger 初始化只依赖配置，不依赖数据库等其他资源
- 返回显式 logger 实例，后续由上层装配
- 后续业务层统一通过注入的 logger 使用，不依赖包级默认 logger

### `pkg/database`

建议职责：

- 根据 `Config` 初始化 GORM MySQL 客户端
- 根据 `Config` 初始化 Redis 客户端
- 统一设置连接池参数
- 启动期执行连接可用性检查
- 返回聚合资源对象与关闭方法

设计原则：

- 初始化失败立即返回错误，不吞错、不延迟
- 不在此阶段引入 Repository 抽象
- 不在此阶段引入事务管理器、读写分离或多数据源

## 实施步骤

1. 初始化 `go.mod` 并引入阶段一所需依赖
2. 建立 `configs/` 目录及 `config.yaml` 样例结构
3. 建立 `pkg/config`，完成强类型结构与 `Load(path)` 入口
4. 建立 `pkg/logger`，完成 JSON logger 初始化与文件轮转策略封装
5. 建立 `pkg/database`，完成 MySQL / Redis bootstrap 和关闭能力
6. 为三大包补充最小必要测试与连接配置校验
7. 在阶段四再统一接入 `cmd/server/main.go`

## 风险与回滚

### 风险

- `lumberjack` 默认行为并非严格自然日切割，需要额外调度逻辑保证目标达成
- MySQL / Redis 的 fail-fast 会使错误在启动阶段集中暴露，要求配置模板必须清晰
- 如果阶段一提前引入过多抽象，会影响后续业务演示部分的可读性

### 回滚策略

- 若跨日主动轮转方案复杂度高于预期，可暂时回退为“大小轮转 + 天数保留”，但需在文档中明确该偏差
- 若聚合资源对象设计不利于后续扩展，可在不改变上层调用顺序的前提下回退为分模块 bootstrap
- 若配置结构出现冗余字段，可优先收缩字段数量，而不是改为全局单例

## 验证清单

- `pkg/config` 是否只负责配置解析，而不做资源初始化
- `Config` 是否包含 `App`、`MySQL`、`Redis`、`Log` 四个子结构
- 配置是否支持“YAML 为主，环境变量覆盖”的加载方式
- `pkg/logger` 是否固定输出 JSON 格式
- 文件日志方案是否基于 `lumberjack`
- “按天切割”是否通过跨日主动 `Rotate()` 方案明确落地
- `pkg/database` 是否统一初始化 MySQL 与 Redis
- 启动阶段是否执行连接可用性检查并采用 fail-fast
- 资源初始化结果是否能被后续 `cmd/server/main.go` 清晰装配
- 本阶段是否未越界实现路由、中间件或业务代码
